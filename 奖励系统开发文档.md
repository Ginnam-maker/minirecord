## 1. 系统架构概述

奖励系统分为两个核心阶段：**规则解析（Rule Parsing）** 与 **日志匹配（Log Matching）**。



---

## 2. 功能详细设计

### 2.1 规则解析 (Rule Parsing API)
**目标**：将用户输入的自然语言（如“每天读书超过5分钟加1分”）转换为结构化数据。

* **输入**：`String` (用户规则)
* **输出**：`JSON Object`

**推荐的 JSON 结构**：
```json
{
  "rule_id": "uuid",
  "original_text": "读书>5分钟加1分",
  "intent": "reading",          // 行为意图
  "condition": {
    "metric": "duration",       // 计量维度：时长/次数/完成度
    "operator": ">=",           // 逻辑算子
    "value": 5,                 // 阈值
    "unit": "minutes"           // 单位
  },
  "reward": 1,                  // 积分变动
  "flexibility": "lenient"      // 匹配策略：lenient(宽容)/strict(严格)
}
```

### 2.2 日志匹配 (Log Matching API)
**目标**：判断用户的每日记录是否命中了已设定的规则。

* **输入**：
    1.  当日记录文本（如：“读了一会书”）
    2.  该用户的所有规则列表（JSON 数组）
* **输出**：命中的规则 ID 及对应的积分变动。

## 3. Prompt 提示词工程 (核心逻辑)

这是解决你“读了一会书”算不算“5分钟”的关键。

### 角色设定
> 你是一个习惯养成助手的规则判定引擎。你的任务是分析用户的随手记日志，并判断其是否达标。

### 处理逻辑 (针对模糊语义)
在 Prompt 中加入以下逻辑引导：
1.  **显式匹配优先**：如果日志中包含明确数值（如“读了10分钟”），严格对比规则阈值。
2.  **隐式语义推断**：如果日志中只有行为（如“读了一会”），且该规则的 `flexibility` 为 `lenient`，则：
    * 判断该行为通常是否暗示了达成。例如，“读了一会”通常意味着完成了基础行为，**判定为命中**。
    * 只有当语境明显未达标（如“翻了一页书就放下了”）时，判定为**不命中**。
3.  **置信度输出**：让模型返回命中概率，以便前端做微交互。

## 4. 数据库设计 (Local/Server)

如果你是纯本地开发，可以在 SQLite 中增加如下表：

| 表名 | 字段 | 说明 |
| :--- | :--- | :--- |
| **Rules** | `id`, `user_id`, `parsed_json`, `status` | 存储解析后的结构化规则 |
| **PointsLog** | `id`, `date`, `rule_id`, `change`, `reason` | 记录积分获取历史 |
| **UserStats** | `user_id`, `total_points` | 用户当前总积分 |

## 5. 开发实现建议 (uni-app 侧)

### 第一步：规则解析时机
不要在每次记账时才解析规则。
* 用户**新增规则**时，调用一次 API，将自然语言转为 JSON 存储在本地。
* 这样在记录匹配时，LLM 只需要面对“固定的 JSON”而不是“未知的文字”。

### 第二步：日志匹配请求示例
你可以一次性将当天的“一句话”和“5条规则”发给 LLM，让它一次性返回结果，节省 API 成本。

```javascript
// 伪代码示例
async function matchUserLog(dailyLog, userRules) {
  const response = await callLLM({
    task: "MATCH_REWARDS",
    content: dailyLog,
    rules: userRules // 传入解析后的 JSON 规则
  });
  
  if (response.matches.length > 0) {
    applyPoints(response.matches);
    showToast(`触发奖励：+${response.totalPoints} 积分`);
  }
}
```

### 第三步：UI/UX 微交互
针对你提到的模糊匹配，建议在 miniRecord 的记录列表下方，给被模型“猜中”的积分加一个虚线边框。
* 用户点击虚线可以手动撤销：“我其实只读了1分钟，不该加分”。
